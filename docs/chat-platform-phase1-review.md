# Final review — `docs/chat-platform-phase1.md`

## 1. Verdict

**Approve with changes.** The core/platform/surface/plugins decomposition and the tests-first step ordering are right, but the plan as written is not executable: three load-bearing architecture pieces are missing (transport interface, host contract, client/server tool split), the per-step "four layers green" gate cannot currently be run, and the standing rules have zero enforcement mechanism. All are fixable as plan amendments before Step 0 — none require abandoning the approach.

## 2. What the plan gets right

- **The split is sound.** core/platform/surface/plugins is the correct decomposition for R4/R5, and the order rationale (tests → contracts → state → UI → plugins) is correct: contracts before extraction means code moves once.
- **Step 0 as a safety net** is the right instinct for R2's no-regression contract, and Step 5's one-commit-per-plugin pattern is the right granularity.
- **5e (attachments/voice stay core)** is a correct boundary call — they're chat-shaped, not dashboard-shaped.
- **Reusing KodyChat for the brand surface** (R3) is preserved: Step 6's "ChatSurface + branding plugin + minimal grant" is a thin wrapper, not a new UI.
- The already-well-factored pieces (`kody-chat-reducer`, `kody-chat-live-session`) mean the state layer is closer to done than the plan fears — the risk is concentrated in `sendText` and the UI split, and the plan's ordering at least isolates those.

## 3. Required changes before starting (ordered by severity)

### HIGH

**H1. Define a `ChatTransport` interface — the real monolith is `sendText`, not the file.**
The target tree says "core: … streaming, transport" but never defines the interface, and that interface is the whole game: `sendText` is a single ~1,630-line callback (KodyChat.tsx:2677–4307) interleaving three incompatible protocol families (Brain sync-SSE, kody-direct client-driven tool loop + `preview_act` chaining, engine via GH Actions trigger) with direct `setMessages`/`toast`/`flushSync`/`router.push` calls. Without the interface, Step 2 either fails or relocates the monolith.
**Edit:** Add a deliverable to Step 1 (or a Step 1.5): (a) `ChatTransport` — `send(turn, ctx)`, an emitted `ChatEvent` union (token/message/tool-call/tool-result/directive/status/error/done), `abort(sessionId)`, restore/status probing; (b) three adapters, each with its own lifecycle model (brain = server-stateful pinned chat id; kody-direct = client tool loop; kody-live = reducer-driven runner lifecycle) — do not force one reducer over all three; (c) directives (dashboard-navigate, SwitchAgent) modeled as ChatEvents the surface interprets, keeping router/toast out of core. Step 2 then extracts `sendText`'s branches into these adapters, unit-tested against recorded event streams.

**H2. Extend the `ChatPlugin` contract with the hooks the target features actually use.**
None of terminal/commands/vibe/goals is primarily "slots + tools": slash commands are send middleware (`expandSlashCommand`, KodyChat.tsx:4845); terminal is a send interceptor (`parseKodyTerminalIntent`, :4842) + per-session persisted state + an alternate surface; goals need host data (knownGoals) and emit host effects (`onDirectToGoal` → scope change); vibe force-overrides terminal mode (`vibeMode ? "ai" : terminalRegistry.mode`, :1322).
**Edit:** Step 1's contract adds: (a) ordered composer/send middleware (transform/consume outgoing text) with deterministic registry ordering plus a test pinning today's precedence (terminal intent before slash expansion); (b) per-session plugin state with plugin-owned persistence key/schema; (c) a host-context read channel and an effect/dispatch channel to the host; (d) a surface-mode arbitration concept in core (plugins declare exclusive display modes; platform resolves precedence) so vibe suppressing terminal never becomes a plugin→plugin import. Step 4 adds the middleware mount point; Step 0's regression specs cover the middleware behaviors (goal mention consumes message, terminal intent reroutes, `/init` bypasses the LLM).

**H3. Specify the client/server split for `registerTools()` — and admit the default backend doesn't get plugin tools in phase 1.**
Tools execute server-side: the in-process route (app/api/kody/chat/kody/route.ts, ~1,361 lines) assembles ~30 tool factories with server context; a client-registered plugin cannot inject them. Brain's tools live on an out-of-repo server; engine tools arrive via MCP/`mcpServers` config in a GH Actions runner. And the admin default agent is `kody-live` — so as written, phase 1 ships plugin tools the default admin path never sees.
**Edit:** Step 1: split the contract into a client manifest (slots, capabilities, theme) and an optional server module exporting tool factories (`(serverCtx) => tools`) registered in a server-side registry imported by the kody route from a server-only entry point (plugin tool handlers must never enter the client bundle — verify with the existing clean-build rule). Step 4: int tests exercise the real kody route handler with a fixture server-half plugin, and every registered tool requires a zod input schema (preserving what `app/api/kody/chat/tools/*` already does right). Replace "engine/brain pass-through later" with the actual mechanisms (engine = MCP config generation; brain = out-of-repo protocol change) and explicitly scope both OUT of phase 1, stating that kody-live won't see plugin tools yet.

**H4. Add a "Host contract" section — ChatRailShell and its dual mount are absent from the plan.**
ChatRailShell mounts KodyChat **twice** (desktop rail :412, mobile sheet :519; desktop hidden-not-unmounted on mobile) and today the instances converge only via localStorage. A module-level core store silently merges them (behavior change, violating the plan's own invariant); a per-instance store constrains the registry design. The `ChatRailApi` prop surface (:53–99: scope, composerInjection, attachmentInjection, previewContext, onIssueCreated, lockedAgentId, vibeMode, presentation, hideTerminalMode, railFullscreen) is consumed app-wide and fed by the out-of-repo element-picker extension.
**Edit:** New section stating: (1) ChatRailShell is the host, stays outside the platform, and its prop surface is a frozen external contract for phase 1; (2) explicit decision in Step 1/2: core stores + registry are instantiated **per KodyChat instance** (plugin definitions global as pure data, slot/tool instantiation per mount) — collapsing the dual mount is its own deliberate step with its own e2e, never a side effect; (3) Step 3 preserves the injection path and the stable-identity dispatcher (ChatRailShell.tsx:313–318) through the Composer extraction, with an int test driving ChatRailApi injection → chip renders; (4) Step 0 adds a spec covering simultaneous desktop+mobile mounts; (5) Step 6 lists `ClientChatSurface`'s current props (presentation, hideTerminalMode, railFullscreen) as inputs the capability grant must reproduce. Fix the stale "mounted once" comments while in there.

**H5. Define the e2e execution contract — the per-step gate is currently unenforceable.**
`playwright.config.ts` sets `webServer: undefined` and targets a deployed BASE_URL; `.env` pins BASE_URL to the Vercel URL; 23 of 34 specs self-skip without `E2E_GITHUB_TOKEN`. As written, "all four layers green per commit" degrades to unit+int, or validates the *previous* commit's deploy — for a refactor whose entire safety story is the e2e contract.
**Edit:** Step 0 encodes: (a) a `test:e2e:local` script (or `webServer` block) that starts `pnpm dev` on :3333 and forces `BASE_URL=http://127.0.0.1:3333`, overriding `.env`; (b) the Step-0 regression/contract suite is **fully mocked and token-free so a skip is impossible** — token-gated specs (chat-live-flow, chat-rail-smoke) are optional live layers, not the gate; (c) the per-commit gate is unit + int + smoke + a chromium-only chat-scoped e2e subset (`tests/e2e/chat-* admin-chat-regression.spec.ts --project=chromium`), with the full two-project run reserved for milestone steps (0, 3, 7) — drop mobile-chrome from the per-commit gate since chat specs skip on mobile; (d) deployed-URL runs are a backstop, not the gate. Reconcile the port-3399 default in chat-kody-attachments.spec.ts.

**H6. Add real enforcement for the standing rules — the current lint config contradicts three of them.**
`eslint.config.mjs` has no `no-console`, `no-explicit-any` is `warn` (never fails), no `max-lines`, and no import-boundary tooling. The most load-bearing rule (core never imports surface; plugins never import each other) is the least enforced, and this codebase's history shows convention-only boundaries erode (KodyChat itself; existing inversion: `useChatTerminalRegistry.ts:16` imports `authHeaders` from components/).
**Edit:** Land in **Step 1**, not Step 7: (1) an eslint override scoped to `src/dashboard/lib/chat/**` with `no-console: error`, `no-explicit-any: error`, `max-lines: ["error", 800]`; (2) `import/no-restricted-paths` zones (eslint-plugin-import is already installed): core may not import surface/plugins/components; plugins may import platform+core, never sibling plugins; surface not importable by core; (3) blocking lint (`--max-warnings=0` or error-level rules) wired into every step's gate. Step 7 then just verifies the config covers all plugin dirs. Step 2/3 note: eliminate the existing `authHeaders` inversion by moving it into chat/core.

**H7. Move locale scaffolding to Step 1 — the i18n standing rule currently depends on a layer built four steps later.**
The repo has zero UI-locale infrastructure, yet the rule requires every extracted string to go through the locale layer from Step 1, while the layer arrives at 5.5 — after Step 3 extracts essentially every string in the 6,953-line component. As ordered: dead-letter rule or double-touch of every file.
**Edit:** Split 5.5: the minimal catalog + provider (en-only, per-plugin namespaces, `t()` helper) lands in Step 1 so Steps 2–5 extract strings into the catalog in one touch. Keep the rest of 5.5 where it is (brand `locale`, surface-root `dir`, logical-CSS audit, RTL e2e extending the existing text-direction specs), plus one added line: surface-root `dir` must not override the existing per-message direction logic (`getMessageDirection`, KodyChat.tsx:471–490) — assert per-message dir survives in the RTL e2e.

**H8. Re-scope Steps 2 and 3 into sub-commits and fix their false premises.**
`kody-chat-reducer.ts` and `kody-chat-live-session.ts` are **already** extracted pure modules — relocating them is a path move, while the actual hard work (session persistence with module-level debounce maps, REHYDRATE_RESTORED ordering guards at KodyChat.tsx:1043–1150, the `sendText` glue) shares the same single commit. Step 3 then splits ~6k lines in another single commit with a mathematically unreachable ≤800 average while plugin UI is still inside.
**Edit:** Step 2 → 2a (relocate the two pure modules, imports only), 2b (extract session persistence + rehydration ordering, with a unit test pinning the mount-ordering contract), 2c (extract streaming glue into the H1 transport adapters; send→stream→persist int test lands here). Also fix: "MessageList (wraps `MessagesView`)" is wrong — `MessagesView` is the /messages team-messaging surface; MessageList is extracted from KodyChat's inline transcript block.

**H9. Audit the ~18 source-text unit specs before Step 2 — they will all break, or worse, mislead.**
At least 17 specs in tests/unit/chat/ (plus ~5 elsewhere) `readFileSync` KodyChat.tsx and assert on source text (e.g. counting `<div>` nesting). Steps 2–3 mechanically fail all of them even with identical behavior, and their green status contributes nothing to the no-regression guarantee.
**Edit:** Add a Step 0 task: for each source-text spec, decide **retire** (behavior now covered by the extended e2e regression) or **rewrite** as a behavior test against the new core module. Exception: move `kody-chat-live-session.spec.ts` with its module unchanged — it *is* the localStorage key/migration contract.
